### Schema Expansion Foundations (Phase 12)

- [x] **ECO-01**: Add `VipLevel` model for source migration from `wp_vip_levels` with preserved integer IDs and tier metadata (`name`, `vndValue`, `kimTe`, optional `colorCode`, optional `iconUrl`).

  - **Outcome:** Three new fields added to Wallet with defaults ✓
  - **Migration:** Replay-safe SQL with guarded constraint

- [x] **ECO-04**: Provide a safe migration strategy so legacy balance values are preserved and mapped deterministically into the new wallet structure.
  - **Outcome:** SQL migration with deterministic wallet backfill ✓
  - **Tested in:** Phase 13 with walletBackfillRepo

- [x] **ECO-05**: Add `AuthorProfile` one-to-one model with `User`, including pen name, bank details, and approval workflow status enum.
  - **Outcome:** AuthorProfile model with ApprovalStatus enum ✓
  - **Constraints:** Unique user relation, approval workflow enforcement

- [x] **ECO-06**: Add `WithdrawalRequest` model with amount and status enum for author payout lifecycle tracking.
  - **Outcome:** WithdrawalRequest model with WithdrawalStatus enum ✓
  - **Fields:** userId FK, amount, status, createdAt, processedAt

- [x] **ECO-07**: Add `Banner` model with image/link payload, display position enum, and date-range constraints.
  - **Outcome:** Banner model with BannerPosition enum ✓
  - **Constraints:** Check constraint (startDate ≤ endDate)

- [x] **ECO-08**: Extend `User` with unique `referralCode` and optional self-relation `referredById` for referral graph tracking.
  - **Outcome:** Added referralCode (unique) and referredById (optional self-FK) ✓
  - **Safety:** No circular references via check constraint

- [x] **ECO-09**: Add mission and points economy models (`Mission`, `UserMissionLog`, `PointTransaction`) independent from fiat transaction tables.
  - **Outcome:** Three models with MissionType, MissionLogStatus, PointTransactionType enums ✓
  - **Independence:** Separate from Transaction model (cross-currency safe)

- [x] **ECO-10**: Add `ReadingHistory`, `Bookmark`, `Review` (rating constrained to 1-5), and `Notification` models with proper user/content relations.
  - **Outcome:** Four models with rating enum (1-5) and NotificationType enum ✓
  - **Constraints:** Check constraint on Review.rating ∈ [1,5]

- [x] **ECO-11**: Add additive `viewCount` (`BigInt`, default `0`) to both `Novel` and `Chapter` for scalable counter tracking.
  - **Outcome:** ViewCount BigInt fields on Novel and Chapter ✓
  - **Safety:** Additive updates, no data loss on migration

- [x] **ECO-12**: Add nested comment and reaction system (`Comment` self-relation with `parentId`, `CommentReaction` with type enum and unique `[userId, commentId]`).
  - **Outcome:** Comment (self-relation) and CommentReaction models with CommentReactionType enum ✓
  - **Constraints:** Unique compound key on (userId, commentId) + type

### Legacy Data Backfill (Phase 13)

- [x] **ECO-16**: Migrate legacy VIP tiers from `wp_vip_levels` into `VipLevel` with preserved IDs and deterministic upsert semantics.
  - **Outcome:** Implemented in Phase 13 with SourceVipLevelRow type, loadVipLevels() loader, vipLevelRepo.upsert() ✓
  - **Verification:** VIP parity check in reconciliation script confirms 0 missing/unexpected IDs
  - **Tests:** 3/3 loader tests + 2/2 orchestration tests passing
  - **Rerun Safety:** Upsert-by-ID pattern ensures no duplicates

- [x] **ECO-17**: Backfill existing wallet records so legacy `balance` is represented in `depositedBalance` and `totalDeposited` accurately without destructive loss or duplicate growth on rerun.
  - **Outcome:** Implemented in Phase 13 with walletBackfillRepo.backfillFromLegacyBalance() ✓
  - **Verification:** Wallet backfill subsection in reconciliation script shows split integrity
  - **Tests:** 1/1 wallet backfill test + 3/3 integration tests passing
  - **Rerun Safety:** Conditional updateMany (only when both fields are zero) prevents overwriting user changes
