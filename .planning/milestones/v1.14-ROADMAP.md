## Overview

## Phases

### Phase 35: Frontend Local Authentication

Status: Planned

Goal: Build responsive login and registration UI forms with client-side validation, error handling, and session integration.

Requirements: AUTH-01, AUTH-02

Scope:
- Responsive login form (username/email + password, client-side validation).
- Registration form with password strength validation and duplicate detection.
- Error messaging for invalid credentials and network failures.
- Session state integration with existing token management.
- commic_session cookie storage and bearer token support.

Success Criteria:
- User can login with username or email, receives token and cookie immediately.
- User can register with email/username uniqueness validation and auto-login on success.
- Password strength meets complexity requirements (8+ chars, letter + number).
- Duplicate email/username returns clear, actionable error messages.
- Forms display client-side validation errors before submission.

Status: Planned

Requirements: ACCOUNT-01

Success Criteria:
- User can view their current profile information.

### Phase 40: Purchase Segmentation

Status: Planned

Goal: Separate individual chapter purchases from combo purchases so admins can monitor each monetization stream independently.

Requirements: MONITOR-01, MONITOR-02

Scope:
- Backend: split purchase history into separate chapter and combo endpoints with typed response contracts.
- Frontend: replace single purchase table in dashboard with two labeled sections (individual chapter purchases + combo purchases).
- Chapter purchase stream retains existing pagination behavior.
- Combo purchase stream sourced from `COMBO_PURCHASE` transaction ledger rows.

Success Criteria:
- `GET /finance/history` returns chapter-only purchases with no combo rows mixed in.
- `GET /finance/combo-history` returns combo purchases independently.
- Dashboard displays both streams as separate labeled tables.
- Admins can distinguish purchase types at a glance from section headings and type-specific columns.

Plans:
- [ ] 40-01-PLAN.md - backend purchase-history segmentation (TDD)
- [ ] 40-02-PLAN.md - dashboard split purchase monitoring UI

### Phase 41: Author Dashboard & Content Management UI

Status: Planned

Goal: Build protected author/admin dashboard workflows for managing novels and chapters with authenticated API integration.

Requirements: AUTHORDASH-01, AUTHORDASH-02, AUTHORDASH-03, AUTHORDASH-04

Scope:
- Protected /dashboard/author route restricted to AUTHOR and ADMIN roles.
- Novel CRUD management UI (list, create/edit form, delete confirmation).
- Chapter CRUD management UI per selected novel with chapterNumber support and ordered display.
- Authenticated API integration to existing /novels and /chapters endpoints with bearer token wiring.

Success Criteria:
- AUTHOR and ADMIN users can open /dashboard/author while unauthorized users are redirected.
- Novel list/create/edit/delete flows work end-to-end with confirmation on destructive actions.
- Chapter list/create/edit/delete flows work end-to-end and preserve chapterNumber ascending order.
- Frontend mutations to protected content endpoints always include session bearer token.
