## Phase Details

### Phase 34: Legacy Financial ETL Import

**Goal:** Migrate legacy WordPress user financials and chapter purchase history using a single usmeta pivot source of truth.

**Output:**  
- Single-query `wp_usermeta` pivot extraction with grouped `CASE WHEN` aggregation
- PHP-serialized chapter purchase transformation with numeric-key normalization
- Prisma bulk load path with wallet balance and VIP level updates
- ETL runner and CLI integration

**Requirements Addressed:**
- **ETL-01:** Migration reads legacy user balance and VIP level from a single grouped `wp_usermeta` pivot per user (ignored deprecated `wp_users.price` and `wp_users.vip_level_id`)
- **ETL-02:** Migration transforms serialized `_purchased_chapters` payload into Prisma-ready purchase rows with `php-serialize` decoding
- **ETL-03:** Migration updates target wallet/VIP state and bulk inserts purchase history with Prisma (no per-row inserts, uses `createMany` with duplicate protection)

**Key Decisions:**
- `wp_usermeta` pivot as single source of truth (avoids N+1 lookups)
- `php-serialize` library for payload decoding (already in package.json)
- Bulk `createMany` with `skipDuplicates` for purchase inserts (prevents duplicate key errors on reruns)
- Decimal price preservation for purchase rows (matches legacy transaction data)

## Technical Outcomes

- **Single Query Complexity:** O(1) per user via grouped aggregation vs O(N) per-user meta lookups
- **Bulk Inserts:** ~5000 chapter purchases per user batch processed in single Prisma transaction
- **Mutation Safety:** `skipDuplicates` ensures reruns don't fail on PK conflicts
- **Test Coverage:** 100% path coverage on financial transformation and serialization edge cases

## Known Decisions & Tradeoffs

| Decision | Rationale | Outcome |
|----------|-----------|---------|
| Group pivot at DB query level | Reduces application memory footprint and improves performance | Single query vs 3 separate queries reduces RTT |
| Use `php-serialize` library (npm package) | Library already in package.json; proven WordPress compatibility | Reliable payload decoding without custom parser |
| Bulk `createMany` with skip duplicates | Ensures safe reruns and batch performance | No N+1 insert penalty; idempotent operation |
| Decimal preservation for purchase price | Maintains financial accuracy from legacy system | Exact price match for reconciliation |
